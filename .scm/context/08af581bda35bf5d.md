# bacon: Background Rust Checker

Continuous error/warning display alongside editor.

## Commands
```bash
bacon              # check
bacon test/clippy/check-all/nextest
bacon --path ../proj --job custom
```

## Keys
`t`=tests `c`=clippy `d`=docs `f`=failing-only `esc`=back `ctrl+j`=jobs `h`=help `q`=quit

## Config: bacon.toml (project, version control)
```toml
default_job = "clippy"

[jobs.clippy-all]
command = ["cargo", "clippy", "--all-targets", "--", "-D", "warnings"]

[jobs.check-win]
command = ["cargo", "check", "--target", "x86_64-pc-windows-gnu"]
```

**Job props:** `command`(req), `analyzer`("standard"), `allow_warnings`(false), `background`(true), `watch`(src), `on_success`, `ignore`

## Config: prefs.toml (global)
`bacon --prefs` - keybinds, display, sounds, skins

## Load order (later wins)
defaults → global prefs → `BACON_PREFS` → workspace Cargo.toml → workspace bacon.toml → package configs → `BACON_CONFIG` → `--config-toml`

## Test Layout
Unit: inline `#[cfg(test)]` | Integration: `tests/*.rs` | Acceptance: `tests/acceptance/features/*.feature`

## Patterns
```rust
// Logging constants
pub const USER_CREATED: &str = "user_created";
info!(message = logmsg::USER_CREATED, username = %username);

// Error constants
pub mod errmsg { pub const DIVIDE_BY_ZERO: &str = "cannot divide by zero"; }
#[error("{}", errmsg::DIVIDE_BY_ZERO)] DivideByZero,

// IoC via generics

```

## Best Practices
- Memory: `Arc<RwLock<T>>` read-heavy, `Arc<Mutex<T>>` write-heavy, accept `&str`
- Concurrency: thread-safe, minimal lock time, consistent order, document
- Errors: `Result<T,E>` + `?`, NO `unwrap()`/`expect()` in libs
- Style: `cargo fmt`, fix clippy, `///` docs explain WHY

# bacon: Background Code Checker

Continuous code checker running alongside editor, showing errors/warnings in real-time.

## Quick Reference

```bash
bacon              # cargo check
bacon test         # tests
bacon clippy       # linter
bacon check-all    # all targets
bacon nextest      # nextest runner
bacon --path ../other-project
bacon --job my-custom-job
```

## Keys

| Key | Action |
|-----|--------|
| `t` | tests |
| `c` | clippy |
| `d` | cargo doc |
| `f` | restrict to failing test |
| `esc` | previous job/all tests |
| `ctrl+j` | list jobs |
| `h` | help |
| `q` | quit |

## Config: bacon.toml (project)

Init: `bacon --init`. Version control this file.

```toml
default_job = "clippy"

[jobs.clippy-all]
command = ["cargo", "clippy", "--all-targets", "--", "-D", "warnings"]
need_stdout = false

[jobs.check-win]
command = ["cargo", "check", "--target", "x86_64-pc-windows-gnu"]

[jobs.check-examples]
command = ["cargo", "check", "--examples"]
watch = ["examples"]
```

### Job Properties

| Property | Purpose | Default |
|----------|---------|---------|
| `command` | Command tokens | Required |
| `analyzer` | Output parser | `"standard"` |
| `allow_warnings` | Warnings=success | `false` |
| `background` | Run in background | `true` |
| `watch` | Extra dirs to monitor | `src` |
| `on_success` | Success action | None |
| `ignore` | Glob excludes | None |

## Config: prefs.toml (global)

Generate: `bacon --prefs`. Contains keybindings, display settings, sound/notifications, color skins.

## Config Load Order (later overrides earlier)

1. Internal defaults
2. Global `prefs.toml`
3. `BACON_PREFS` env
4. Workspace `Cargo.toml` metadata
5. Workspace `bacon.toml`
6. Package-level configs
7. `BACON_CONFIG` env
8. `--config-toml` arg

## Tips

- Errors before warnings; most recent last
- Runs on file save automatically
- Works with small terminals
- Use `watch` for non-src directories

I've read and understood the "Lean on the Compiler" fragment. This is a development practice about using compiler/LSP errors as a guide when making changes:

**Core pattern:** Make breaking change → build/save → fix errors shown → repeat until green

**Key points:**
- Compiler catches ALL usages (manual search misses some)
- Errors become your TODO list
- LSP provides real-time feedback without explicit build
- Works for: renames, signature changes, type changes, deletions, field changes

**Anti-patterns to avoid:**
- Manual grep/search for usages
- Adding shims to avoid breaks
- Commenting out code to test
- TODO comments for "later"

**Mindset:** Breaking changes are features, not problems. Trust tooling, make bold changes, chase red squiggles.

This complements the sibling `prototype` and `reductive` fragments - all support rapid, confident iteration.

# Prototype Mode

Build correctly. No compromise.

## Core Rules

- Delete deprecated code immediately—no preservation
- Wrong code → delete and rebuild correctly
- Bad format/API/behavior → replace entirely, don't wrap
- "This breaks X" → fix X
- Migration = someone else's problem

## Hard Changes Only

1. Delete offending code
2. Rebuild correctly
3. Fix everything that breaks
4. Never add shims/flags/fallbacks

## In Practice

Rename/correct/restructure/remove freely—fix all refs, callers, consumers, handlers.

## Forbidden

- `// Deprecated` or `@deprecated`—delete now
- Feature flags for old behavior
- Version checks/legacy conditionals
- Old→new wrappers
- Defaults preserving old behavior
- Union types for old+new formats
- "Backwards compatibility" comments
- Fallback logic for renames
- Multi-version handling
- "Handle both formats" when you control data

## Standard

One version: the correct one. No compat. No migration period. No deprecation cycle. Only correct implementation, now.

# Reductive Development (Compressed)

**Greenfield/prototype only** - production needs backwards-compat judgment.

## Process: Reduce BEFORE implementing significant tasks

1. Pause - don't start immediately
2. Scan: duplicate funcs/patterns, copy-paste, multiple impls
3. Target: dead code, unused imports/vars/params, over-engineering, wrappers, compat shims
4. Clean first, then proceed on smaller codebase

## Rules
- Keep tests green & coverage
- Smaller = better
- Delete > deprecate

## Why
Fewer bugs, less to understand, prevents dup spread, task may simplify, tech debt compounds

## Apply: new features, multi-line fixes, refactors, multi-file changes
## Skip: typos, comments, single-line, docs-only

# Reductive Development

**Disclaimer:** For greenfield/prototype projects only. Use judgment on backwards compatibility for production systems.

Reduce codebase before starting significant tasks.

## Pre-Task Process

1. **Pause** - Don't implement immediately
2. **Scan for duplication** - duplicate functions, similar patterns, copy-pasted logic, multiple implementations
3. **Identify reduction targets** - dead code, unused imports/variables/params, over-engineered abstractions, wrapper functions, compatibility shims
4. **Reduce first** - clean up before starting task
5. **Proceed** - work on smaller codebase

## Rules

- Preserve test coverage
- Keep tests green
- Smaller = better
- Delete over deprecate

## Why

- Fewer bugs in smaller codebases
- Less to understand before changes
- Prevents duplication spread
- Task may simplify post-reduction
- Tech debt compounds

## Apply To

New features, multi-line bug fixes, refactoring, multi-file changes

## Skip For

Typos, comments, single-line changes, docs-only

## No Legacy Accommodation

- Bad format → new format, don't support both

- Rename correctly, fix all refs

- `// Deprecated` comments — delete now
- `@deprecated` annotations — delete code
- Feature flags for old behavior
- Version checks/legacy conditionals
- Old→new wrapper functions
- Defaults preserving old behavior
- Union types for old+new formats
- Any "backwards compatibility" comments
- Fallback logic for renames
- Multi-version case handling
- "Handle both formats" logic when you control data generation

## Standard

One version: correct one.

No v1 compat. No migration period. No deprecation cycle. Only correct implementation, built correctly, now.

Writing code for "the old way"? Stop. Delete it. Only the new way exists.
